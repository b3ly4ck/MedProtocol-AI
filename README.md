# MedProtocol Core: Спецификация Backend-фич (V2.2)

## Итерация 1: Гибкие структуры и интеллектуальная проверка

### 1. Объектно-ориентированный конструктор протоколов (Protocol Factory)
**Задача:** Реализовать масштабируемую систему создания кастомных медицинских протоколов, где каждый протокол является динамически собираемым объектом.

* **Техническая реализация:**
    * Применение паттерна **Composite** для описания структуры: `Protocol` -> `Section` -> `Field`.
    * Реализация базового абстрактного класса `BaseMedicalField` с методами `serialize()`, `validate_schema()` и `Youtube()`.
    * Динамическая генерация классов Pydantic в рантайме (через `pydantic.create_model`) на основе конфигурации, собранной врачом. Это обеспечит автоматическую генерацию OpenAPI документации и строгую типизацию эндпоинтов в FastAPI для кастомных форм.
* **Результат:** Единый формат хранения данных в PostgreSQL (JSONB) при полной гибкости состава полей для разных врачебных специальностей с сохранением строгой типизации.

### 2. Инфраструктура LLM-валидации (Prompt & Tool Registry)
**Задача:** Разработать backend-слой для взаимодействия с языковыми моделями (LLM) для автоматического поиска аномалий и ошибок в заполненных данных.

* **Техническая реализация:**
    * **PromptContainer:** Классы для хранения версионированных системных промптов, специфичных для разных типов протоколов.
    * **ToolRegistry:** Набор Python-функций, доступных LLM через Function Calling (например, `check_drug_compatibility(drug_a, drug_b)` или `get_normal_ranges(patient_age)`).
    * **ValidationParser:** Слой парсинга JSON-ответа от LLM, который связывает найденные нейросетью логические ошибки с конкретными UUID полей протокола для подсветки на фронтенде.
* **Результат:** Интеллектуальный ассистент, который не просто проверяет типы данных, а анализирует медицинскую логику введенного анамнеза.

## Итерация 2: Сохранность данных и рабочий стол врача

### 3. Система контроля версий протоколов (Protocol VCS)
**Задача:** Реализовать механизм отслеживания изменений, позволяющий сравнивать версии и восстанавливать предыдущие состояния документа.

* **Техническая реализация:**
    * Применение паттерна **Memento** (Снимок) на уровне базы данных. Каждое сохранение создает иммутабельную запись в таблице `protocol_versions`.
    * Разработка метода `diff()`, возвращающего разницу между состояниями протокола в формате JSON-patch (RFC 6902), чтобы фронтенд мог визуально подсветить, что изменил ИИ или другой врач.
    * Механизм `revert(version_id)` для глубокого восстановления состояния объекта из сериализованного снимка.
* **Результат:** Полная юридическая прослеживаемость изменений (Audit Trail) и защита от некорректных правок ИИ.

### 4. Высокопроизводительный Read-API для рабочего стола врача (Паттерн CQRS)
**Задача:** Разработать оптимизированный движок агрегации и фильтрации для быстрого отображения тысяч сохраненных протоколов в дашборде врача без нагрузки на основной транзакционный контур.

* **Техническая реализация:**
    * Внедрение паттерна **CQRS (Command Query Responsibility Segregation)**: разделение моделей записи (сохранение протоколов) и моделей чтения (списки для интерфейса).
    * Использование **Materialized Views** в PostgreSQL или построение продвинутых GIN/GiST индексов поверх JSONB-полей для мгновенного поиска по вложенным ключам (например, `protocols.data -> 'diagnosis' ->> 'mkb_code'`).
    * Разработка динамического Query Builder API (с использованием `FastAPI-Filter` или `SQLAlchemy-Filter`), который позволяет фронтенду запрашивать протоколы с произвольными комбинациями фильтров (по дате, статусу ИИ-проверки, тегам, значениям конкретных метрик).
* **Результат:** Врач получает мгновенно работающий интерфейс с гибкой фильтрацией (Inbox-style), а база данных не блокируется тяжелыми `SELECT` запросами с множественными объединениями таблиц.

## Технологический стек
* **Язык:** Python 3.12+
* **Фреймворк / Валидация:** FastAPI + Pydantic V2
* **БД:** PostgreSQL (JSONB + GIN Indexes + Materialized Views) + SQLAlchemy 2.0 (Async)
* **LLM:** Интеграция с моделями генерации (API/Local).
