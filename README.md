# MedProtocol AI

## 1. Описание идеи

**MedProtocol AI** — интеллектуальная **система поддержки принятия врачебных решений (СППВР)** в виде веб-интерфейса для заполнения протоколов осмотра пациентов. Текст осмотра обрабатывается с помощью ИИ: выполняется сверка с **клиническими рекомендациями (КР)**, нормализация терминологии и подсказки по **дозировкам лекарственных средств** с опорой на справочники и правила.

Проект адресует две практические задачи:

- **Неструктурированные медицинские данные** — свободный текст протокола приводится к проверяемому, согласованному с КР представлению и структурируется на уровне доменной модели.
- **Снижение риска врачебных ошибок** — автоматическая проверка формулировок, показателей и терапевтических решений относительно актуальных рекомендаций и ограничений по данным пациента.

> **Важно.** Система носит **учебный и вспомогательный** характер; окончательные клинические решения принимает врач, а не программа.

## 3. План разработки

### Итерация 1 — Backend & Core

Демонстрация продвинутого Python в ядре приложения и инфраструктуре данных.

1. **Иерархия медицинских протоколов (ABC + `Generic`)**  
   Абстрактные базовые классы (`abc.ABC`) задают контракты для типов протоколов (например, амбулаторный осмотр, стационарный эпикриз). Параметризация через `typing.Generic` связывает типы полей протокола с конкретными сущностями (пациент, разделы осмотра), что обеспечивает **типобезопасность** на этапе статической проверки и упрощает расширение без дублирования логики.

2. **Асинхронная обработка (`asyncio`)**  
   Пайплайн анализа текста строится на **неблокирующих** вызовах к LLM и внешним **справочникам лекарств** (HTTP/API). Очереди задач, таймауты и параллельное получение справочной информации снижают задержку ответа API и демонстрируют управление конкурентностью в Python 3.

3. **Кастомные дескрипторы (Descriptors)**  
   Для моделей данных реализуются дескрипторы, выполняющие **строгую валидацию** клинически значимых показателей: ЧСС, артериальное давление, ИМТ и др. Инварианты (диапазоны, единицы, согласованность с полом/возрастом при необходимости) инкапсулируются в протокол доступа к атрибутам, без размазывания проверок по всему коду.

**Пример направления архитектуры (упрощённо):**

```python
from abc import ABC, abstractmethod
from typing import Generic, TypeVar

TProtocol = TypeVar("TProtocol", bound="BaseMedicalProtocol")


class BaseMedicalProtocol(ABC):
    @abstractmethod
    def to_structured_payload(self) -> dict: ...


class ProtocolService(Generic[TProtocol]):
    def __init__(self, protocol: TProtocol) -> None:
        self._protocol = protocol
```

### Итерация 2 — AI & Logic

Фокус на поиске по знаниям, наблюдаемости и гибкой бизнес-логике.

1. **RAG по клиническим рекомендациям (PDF)**  
   Документы КР разбиваются, индексируются и извлекаются в контекст генерации: ответы и предупреждения опираются на **фрагменты первичных источников**, а не только на параметры модели.

2. **Метапрограммный слой: декораторы и метаклассы**  
   Декораторы и метаклассы формируют **сквозную инфраструктуру**: автоматическое логирование действий врача (какие разделы изменены, какие запросы к анализу отправлены) и **аудит правок**, предложенных ИИ-агентом (кто/когда принял или отклонил подсказку). Снижается риск «тихих» изменений без следа для учебного и демонстрационного сценария соответствия.

3. **Паттерны «Стратегия» и «Фабрика» (МКБ-10)**  
   Логика проверки протокола (набор правил, приоритет КР, жёсткость предупреждений) выбирается **динамически по коду диагноза**. Фабрика по ключу МКБ-10 возвращает конкретную стратегию валидации/обогащения, что упрощает добавление новых нозологий без изменения общего оркестратора.

## 4. Сторонняя библиотека (Deep Dive): LlamaIndex

**Выбор:** **LlamaIndex** — как основа для RAG над корпусом медицинских документов (в т.ч. PDF с клиническими рекомендациями).

**Обоснование и глубина использования**

- **Векторный индекс** строится поверх чанков текста, извлечённых из PDF; запросы врача и фрагменты протокола превращаются в эмбеддинги и сопоставляются с наиболее релевантными участками КР.
- **Парсинг и структура документов** выходит за рамки «загрузил и векторизовал»: требуется осмысленная настройка **Node Parsers** (размер чанка, перекрытие, разбиение по заголовкам/таблицам там, где это уместно), чтобы не терять клинический контекст и не смешивать несвязанные абзацы.
- **Управление эмбеддингами** — выбор и согласование модели эмбеддингов с размерностью индекса, политика обновления индекса при смене версии КР, контроль стоимости и латентности при масштабировании корпуса.
- **Сложные ретриверы** — комбинация семантического поиска с фильтрами (например, по разделу рекомендаций, году издания), возможные **query transformations** и объединение нескольких источников в единый контекст для LLM — это уже уровень инженерии данных и оркестрации, выходящий за типичный объём вводного курса по Python.

Таким образом, LlamaIndex используется не как «обёртка над одним вызовом API», а как **стек для полного цикла RAG**: ingest → chunk → embed → retrieve → augment → generate, с акцентом на качество извлечения из PDF и воспроизводимость результатов.
